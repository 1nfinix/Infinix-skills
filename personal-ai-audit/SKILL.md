# AI 外脑全景审计

你是用户的本地 AI 协作顾问和个人外脑规划助手。对用户可访问的文件、目录、项目、知识库、配置和工作流进行只读扫描，然后输出三份交付物。

## 设计理念

这不是一次性的提问——是帮助用户理解"我的 AI 外脑能做什么、现状如何、未来怎么演进"的系统性审计。

核心假设：**用户已经和 AI 协作了一段时间，但缺乏全局视图。** 审计的目的就是补上这块。

## 0. 扫描原则

- 默认只读扫描，不修改、不删除、不移动文件
- 优先扫描目录结构、文件名、README、配置摘要、脚本入口、最近输出报告和 Git 状态
- 不要全文读取 .env、密钥、聊天记录、支付信息等敏感文件
- 所有判断必须基于证据，区分「确定观察」和「合理推测」

## 1. 扫描范围（必须覆盖）

### 环境层

获取操作系统、磁盘、内存、Python/Node 版本、AI Agent 版本。

```
# 示例（Hermes Agent 环境）
uname -a && df -h / && free -h
python3 --version && node --version
hermes --version
```

**适配说明**：根据用户使用的 AI Agent 平台，替换对应的版本检查命令。

### 项目层

- 扫描家目录下所有 Git 仓库（`find ~/ -maxdepth 3 -name ".git" -type d`）
- 列出项目目录及大小
- 注意区分「活跃项目」和「归档/废弃项目」

### 知识库层

- 扫描用户的笔记/知识库目录结构
- 统计文件数、大小、覆盖日期范围
- 检查是否有索引/日志/模式文件（index/log/schema）

```
# 示例
find ~/知识库目录 -maxdepth 3 -type f | head -80
find ~/知识库目录 -name "*.md" -mtime -7 -type f  # 最近 7 天活跃度
for d in ~/知识库目录/*/; do echo "$(basename $d): $(find "$d" -type f | wc -l)"; done
```

### 自动化层

- 列出定时任务和调度状态
- 统计自定义脚本数量和类型
- 检查 workspace/项目目录是否规范

```
# 示例（Hermes Agent）
hermes cron list
ls ~/.hermes/scripts/
ls ~/workspace/ 2>/dev/null
```

**适配说明**：使用对应平台的 cron/scheduler 管理命令。

### 技能层

- 统计 Skill/Plugin 数量
- 计算磁盘占用（排除软链接误算）
- 检查是否有休眠/废弃 skill

```
# 示例
find ~/.hermes/skills -maxdepth 2 -name "SKILL.md" | wc -l
du -sh ~/.hermes/skills/
```

### 配置层

- 读取 AI Agent 配置摘要（**不输出密钥**）
- 检查模型/provider 是否完整
- 识别配置缺失或依赖默认值的字段

```
# 示例（Hermes Agent + YAML config）
python3 -c "
import yaml
with open('config.yaml') as f:
    c = yaml.safe_load(f)
m = c.get('model', {})
print('main:', m.get('provider'), m.get('model'))
print('fallback:', c.get('fallback_model', {}).get('model'))
"
```

**注意**：密钥只检查存在/缺失（`if value else 'MISSING'`），**绝不输出实际密钥值**。

## 2. 输出格式

### 2.1 我能帮你做的 10 件事

每件事必须包含：
- **发现线索** — 基于什么文件/配置/行为
- **做什么** — 具体可执行的任务
- **价值** — 对用户的实际收益
- **授权方式** — 需要用户做什么
- **第一步** — 立即可以执行的最小动作
- **优先级** — P0（核心商业）/ P1（已验证）/ P2（建设性）

10 件事应覆盖：内容生产、知识管理、自动化运维、投资/研究、品牌建设、学习成长等维度。

**注意**：必须基于实际扫描证据生成，不可凭空编造。如果没有足够证据支撑 10 件事，有多少列多少。

### 2.2 长期协作计划

分四个时间维度：
- **🔴 立即（本周）** — 紧迫且重要
- **🟡 短期（2 周内）** — 重要不紧迫
- **🟢 长期（1-3 月）** — 战略建设
- **🔁 周期性** — 每日/每周/按需的固定节奏

加上：
- **🚫 协作边界** — 哪些决定必须用户做，哪些 AI 不自作主张

### 2.3 AI 眼中的你

- **确定观察**（基于证据）— 从文件、配置、行为中得出的客观事实
- **合理推测**（基于行为模式）— 从数据中推理但尚未确认的判断，需标注「推测」
- **3 条协作原则** — 基于用户画像提出的协作准则

## 3. 安全边界

- 明确声明本次扫描只读不写
- 声明未读取的内容类型（.env、密钥、聊天记录等）
- API 密钥只检查存在/缺失，不输出值
- 提醒内部路径在公开发布前需脱敏

## 4. 局限声明

- 列出本次扫描无法覆盖的范围（手机本地、第三方平台后台、商业数据等）
- 对不确定的观察标注「推测」而非断言

## 5. 执行约束

- 全部检查应在 60 秒内完成
- 输出直接给用户阅读，面向最终用户而非开发者
- 语言风格匹配用户偏好
- 扫描完成后在报告底部给出「下一步建议」——选择 1 个 P0 项推进

## 平台适配指南

本 skill 的哲学和输出结构是平台无关的，但扫描命令需要根据用户实际使用的 AI Agent 平台进行调整：

| 扫描层 | 通用思路 | 需适配的命令 |
|--------|---------|-------------|
| 环境层 | OS/磁盘/内存/语言版本 | `uname`/`df`/`free` 跨平台，版本命令需适配 |
| 项目层 | Git 仓库+目录大小 | `find`/`du`/`ls` 跨平台 |
| 知识库层 | 笔记目录结构+活跃度 | 需用户告知知识库路径，或从配置中推断 |
| 自动化层 | 定时任务+脚本清单 | 平台特有：`cron list`/`schedule list` 等 |
| 技能层 | Skill/Plugin 清单+占用 | 平台特有：skill 路径和命名约定 |
| 配置层 | 模型/provider 完整性 | 平台特有：配置路径和格式（YAML/TOML/JSON） |

**首次执行前**：先通过环境自检确定用户使用的平台，然后选择对应的命令集。不确定时用通用 Unix 命令兜底。

## 已知坑

- **Config 读取命令可能不存在。** 有些平台的 CLI 没有 `config get` 子命令。备选方案：直接用 Python/YAML 读取配置文件。
- **磁盘占用可能包含软链接/挂载点。** `du` 默认会跟随 symlink。如果某个目录大小异常（如 skills 目录 >100M），优先检查是否包含 `.git` 目录或 symlink 到外部项目。
- **知识库文件数和索引行数不对齐。** 如果索引文件行数远小于实际文件数，说明有孤立文件未索引。需要在报告中标注。
- **触发词冲突。** 如果用户同时有多个审计/诊断类 skill，确保触发词互不重叠，否则命中率不可预期。

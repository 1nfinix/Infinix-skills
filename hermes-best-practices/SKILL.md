# Hermes Agent 实操好习惯清单

> 核心逻辑：配置文件是辅助约束，不是绝对保证。尤其在 Telegram 这种已知存在加载不稳定问题（Issue #5200）的场景下，好习惯的核心是「定期验证、主动沉淀、人工兜底」。

---

## 1. 配置文件维护类

### 1.1 SOUL.md / AGENTS.md 改后验证
改完后新会话开场先让 Agent 复述当前内容和文件路径，确认真的加载了新版本，不要假设生效。

### 1.2 一次只改一类配置
不要在同一时间改多个配置文件再一起验证——出问题时无法定位。一次只改一类，验证通过再改下一类。

### 1.3 改动前留备份
改配置文件前先让 Agent 把当前完整内容贴一遍存到本地，方便改坏了能直接回滚，而不是依赖记忆重写。

### 1.4 MEMORY.md / USER.md 字符预算监控
快接近字符上限时主动清理，而不是等写入报错才发现。定期（比如每周）让 Agent 自己汇报当前这两个文件的字符数和百分比。

**阈值参考**：MEMORY.md 上限 8,000 字符、USER.md 上限 3,000 字符。50% 安全、70% 关注、85% 立即清理。审计方法论详见 `hermes-maintenance` skill 的 `references/memory-audit-methodology.md`。

## 2. 日常委派任务类

### 2.1 危险操作首次验证
涉及删除、发送、资金、生产环境的操作，哪怕规则已写明，第一次真正触发时手动观察 Agent 是否真的停下来确认了。规则文件的效果需要用真实场景验证过一次才能信任。

### 2.2 长任务分阶段汇报
多步骤、长时间的任务中途按阶段汇报进度，不要等到最后才看结果。中间某一步出错越早发现越容易纠正。

### 2.3 指令边界明确
给指令时尽量明确范围和边界，不依赖 Agent「猜」意图。「删除 test 目录下的临时文件」比「清理一下」更安全。

---

## 3. Telegram 场景专属

### 3.1 敏感操作私聊
群聊场景里，敏感或高权限指令尽量私聊单独发，不在群里发。群消息更容易被误读上下文，隐私模式可能导致消息没送达。

### 3.2 定期审查授权用户列表
定期检查 TELEGRAM_ALLOWED_USERS 是否还是期望的人，尤其是团队协作场景人员有变动时。这个列表不会自动同步外部变化。

### 3.3 定期重启网关
网关长时间运行后，建议定期（比如每周）重启一次，尤其是改过 config.yaml 的情况，避免运行中进程和磁盘上的配置不一致。

---

## 4. 记忆和长期信息管理类

### 4.1 重要信息明确存储
不要指望 Agent 会「自己记住」一次性提到的重要信息。如果希望长期保留，明确说「把这个记到 MEMORY.md」。

### 4.2 定期人工审计记忆
偶尔（比如每月）让 Agent 完整输出一遍 MEMORY.md 和 USER.md 内容，自己过一眼过时/错误/不重要的条目，主动人工清理而非任由自动堆积。

### 4.3 敏感信息不进记忆
涉及隐私或敏感信息（健康、财务细节）不要写进 MEMORY.md/USER.md 长期保存，这些文件每次会话被完整读取注入，保留越久暴露面越大。

---

## 5. 信任边界类

### 5.1 确认不是绝对安全网
不要把 Agent 的「确认」当作绝对可靠——尤其在已知 SOUL.md/AGENTS.md 有时不会被正确加载的情况下，重要操作的最后一道防线是人工判断。

### 5.2 定期审视 Hard Constraints
定期（比如每季度）重新审视 SOUL.md 里的 Hard Constraints 部分，看是否还符合当前实际使用场景。规则写完不是一次性的，场景变了规则也该调整。
